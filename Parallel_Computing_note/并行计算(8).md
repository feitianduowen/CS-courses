## 2. 编程标准的作用

课件第 3 页先讲“编程标准的作用”。

### 2.1 规定程序执行模型

编程标准要说明程序按照什么模型执行，例如：

```text
SPMD
SMP
```

SPMD 是多个线程或进程执行同一份程序，不同执行单元根据编号处理不同数据。
SMP 则强调共享内存多处理器上的执行模型。

------

### 2.2 表达并行性

标准要提供语法说明哪些部分可以并行执行。

课件列出：

```text
DOACROSS
FORALL
PARALLEL
INDEPENDENT
```

这些结构的作用是告诉编译器或运行时系统：

```text
这里可以并行执行
这里是并行循环
这里不同迭代之间相互独立
```

------

### 2.3 表达同步

并行程序必须处理同步，因此标准要提供同步机制。

课件列出：

```text
Lock
Barrier
Semaphore
Condition Variables
```

含义分别是：

| 同步机制            | 作用                   |
| ------------------- | ---------------------- |
| Lock                | 互斥访问共享资源       |
| Barrier             | 所有线程到达后一起继续 |
| Semaphore           | 用计数机制控制资源访问 |
| Condition Variables | 条件满足时唤醒等待线程 |

------

### 2.4 获得运行时环境变量

标准还要提供运行时查询接口，例如：

```text
threadid
num of processes
```

也就是：

```text
当前线程是谁？
总共有多少线程或进程？
```

这对于 SPMD 程序非常重要，因为每个线程要根据自己的编号决定处理哪一部分数据。

## 3. 13.1 ANSI X3H5 共享存储模型

### 3.1 背景

课件第 4 页介绍 ANSI X3H5。

它起源于 20 世纪 80 年代中期，当时共享内存并行计算机开始出现，不同厂商都有自己的基于指令的编程环境。

课件提到：

```text
Started in the mid-80’s
shared memory parallel computers
proprietary directive driven programming environments
```

也就是说，X3H5 想解决的问题是：

> 不同共享内存并行机的编程方式不统一，需要一个标准化模型。

------

### 3.2 与 PCF 的关系

课件还提到：

```text
更早的标准化结果—PCF共享存储器并行Fortran
```

PCF 可以看作更早的共享内存并行 Fortran 标准化尝试。

X3H5 则是在这个基础上进一步提出的概念性编程模型。

------

### 3.3 制定时间和语言绑定

课件说明，X3H5 是：

```text
1993年制定的概念性编程模型
```

支持的语言绑定包括：

```text
C
Fortran 77
Fortran 90
```

这说明 X3H5 不是某一种具体语言，而是一个可以绑定到多种语言上的共享内存并行编程模型。

## 4. X3H5 的主要构造

课件第 5 页列出了 X3H5 的核心内容。

### 4.1 并行块：工作共享构造

X3H5 支持几种工作共享结构：

```text
并行块：psections ... end psections
并行循环：pdo ... end pdo
单进程：psingle ... end psingle
```

它们分别对应：

| 构造      | 含义                        |
| --------- | --------------------------- |
| psections | 不同 section 由不同线程执行 |
| pdo       | 循环迭代由多个线程共享执行  |
| psingle   | 只有一个线程执行某段代码    |

------

### 4.2 可嵌套

课件说明这些并行构造可以嵌套。

也就是说，可以在一个并行区域内部再出现工作共享结构。

例如：

```text
parallel
    psections
        section C
        section D
    end psections

    pdo
        F(i)
    end pdo
end parallel
```

------

### 4.3 非共享块重复执行

课件写到：

```text
非共享块重复执行
```

含义是：在并行区域内，如果一段代码没有被 psections、pdo、psingle 这类工作共享结构包围，那么每个线程都会执行一遍。

这和 OpenMP 中 parallel region 内普通语句会被所有线程执行类似。

------

### 4.4 隐式路障和 nowait

X3H5 支持：

```text
隐式路障
nowait
显式路障
阻挡操作
```

通常工作共享结构结束时会有隐式 barrier，保证所有线程完成该结构后再继续。

如果加上 `nowait`，则取消隐式路障，线程做完自己的部分后可以继续向下执行。

------

### 4.5 共享变量和私有变量

课件列出：

```text
共享/私有变量
```

共享变量可以被所有线程访问。
私有变量每个线程有自己的副本。

这对共享内存编程非常关键，因为很多错误都来自变量作用域设置不当。

------

### 4.6 线程同步机制

课件列出四种同步机制：

```text
门插销 latch：临界区
锁：test, lock, unlock
事件：wait, post, clear
序数 ordinal：顺序
```

可以理解为：

| 同步机制 | 用途                   |
| -------- | ---------------------- |
| latch    | 实现临界区             |
| lock     | 显式加锁、测试锁、解锁 |
| event    | 等待和触发事件         |
| ordinal  | 规定执行顺序           |

------

## 5. X3H5 并行性构造示例

课件第 6 页给出一个完整例子，第 7 页用时间图解释线程执行过程。

### 5.1 程序从顺序模式开始

示例开头：

```text
Program main
A
parallel
```

程序最初只有一个基本线程，也叫主线程。
因此 `A` 只由主线程执行。

------

### 5.2 parallel：进入并行模式

执行到：

```text
parallel
```

程序派生出多个子线程，形成一个线程组。

在并行区域中，普通语句会被每个线程重复执行。

例如：

```text
B
```

第 7 页图中，线程 P、Q、R 都执行了 `B`。

------

### 5.3 psections：并行块

示例中：

```text
psections
section
    C
section
    D
end psections
```

含义是：

```text
一个线程执行 C
另一个线程执行 D
```

`end psections` 处有隐式路障，必须等 C 和 D 都完成后才能继续。

第 7 页图中也标出了这里的“隐式路障同步”。

------

### 5.4 psingle：单线程执行

示例中：

```text
psingle
    E
end psingle
```

表示只有一个线程执行 `E`。

执行完后转回并行模式。

这适合只需要做一次的工作，例如初始化、输出、分配共享资源等。

------

### 5.5 pdo：并行循环

示例中：

```text
pdo i = 1, 6
    F(i)
end pdo no wait
```

表示循环 `F(1)` 到 `F(6)` 的迭代由线程组共享执行。

第 7 页图中可以看到：

```text
线程 P 执行 F(1:2)
线程 Q 执行 F(3:4)
线程 R 执行 F(5:6)
```

由于 `end pdo no wait`，这里没有隐式路障，所以线程执行完自己的循环部分后可以继续执行 `G`。

------

### 5.6 end parallel：回到顺序模式

最后：

```text
end parallel
H
```

结束并行区域后，程序回到顺序模式，由主线程单独执行 `H`。

## 7. 什么是线程

课件第 10 页用图说明线程。

### 7.1 线程的定义

课件列出：

```text
General-purpose solution for managing concurrency
Multiple independent execution streams
Shared state
Preemptive scheduling
Synchronization
```

线程可以理解为：

> 一个进程内部的多个独立执行流，它们共享进程的内存、文件等状态，但各自有自己的执行路径。

------

### 7.2 线程共享状态

图中显示多个线程连接到同一个：

```text
Shared state
memory, files, etc.
```

也就是线程共享：

```text
全局变量
堆
文件描述符
进程资源
```

但每个线程通常有自己的：

```text
栈
寄存器状态
程序计数器
```

------

### 7.3 线程需要同步

因为线程共享内存，所以它们很容易同时访问同一个变量。

因此必须使用：

```text
locks
conditions
mutex
barrier
```

否则就会出现数据竞争。

------

## 8. 线程相对进程的特点

课件第 11 页列出线程的优点和注意事项。

### 8.1 共享相同内存空间

线程共享同一进程的地址空间。

优点是通信方便：

```text
一个线程写共享变量
另一个线程可以直接读
```

不需要复杂 IPC。

------

### 8.2 创建开销小

课件比较线程和 `fork()`：

```text
与标准 fork() 相比，线程开销很小
内核无需复制进程内存空间或文件描述符
节省 CPU 时间
```

这说明线程比进程更轻量。

------

### 8.3 可利用多 CPU

课件指出：

```text
和进程一样，线程将利用多 CPU
```

如果程序为多处理器系统设计，线程可以并行运行在多个 CPU 上，适合计算密集型应用。

------

### 8.4 无需复杂 IPC

课件写到：

```text
支持内存共享，无需使用繁琐的 IPC 和其它复杂通信机制
```

这是共享内存编程的最大优点。

但代价是：同步和互斥必须小心处理。

------

### 8.5 POSIX 线程没有父子关系

课件指出：

```text
POSIX 线程标准不记录任何“家族”信息。无父无子。
```

如果要等待一个线程终止，必须保存它的线程 ID，并传给：

```c
pthread_join()
```

线程库不能自动判断你要等待哪个线程。

------

## 9. 线程内存结构图

课件第 12、13 页用图展示线程与进程内存关系。

### 9.1 进程复制与线程共享的区别

第 12 页图中对比了 `fork()` 前后状态。

进程 fork 后，新进程有自己的地址空间副本。

而线程不是复制整个进程空间，而是在同一个进程地址空间中增加执行流。

### 9.2 多线程内部结构

第 13 页图中显示：

```text
多个线程各自有 Register 和 Stack
共享 Text、Data、Heap
```

因此：

```text
局部变量通常在线程自己的栈上
全局变量、静态变量、堆对象通常被线程共享
```

这也是为什么共享变量需要加锁，而局部变量通常不需要加锁。

## 12. Pthreads 计算 π 的实例

课件第 16、17 页给出 Pthreads 实现计算 π 的例子。

### 12.1 主函数做什么

第 16 页红框标出几个关键步骤：

```text
初始化互斥锁 pthread_mutex_init
创建线程 pthread_create
等待线程结束 pthread_join
```

基本流程是：

```text
1. 读取积分区间数和线程数
2. 初始化全局变量
3. 初始化 reduction_mutex
4. 创建 num_threads 个线程
5. 每个线程执行 PIworker
6. 主线程 join 等待所有线程结束
7. 输出 pi
```

------

### 12.2 工作线程做什么

第 17 页中 `PIworker` 的核心是：

```c
for (i = myid + 1; i <= n; i += num_threads) {
    x = w * ((double)i - 0.5);
    sum += f(x);
}
mypi = w * sum;
```

每个线程按照自己的编号 `myid` 做循环划分：

```text
线程 0 处理 1, 1+num_threads, ...
线程 1 处理 2, 2+num_threads, ...
```

这是典型的循环交错划分。

------

### 12.3 用互斥锁做归约

第 17 页红框标出：

```c
pthread_mutex_lock(&reduction_mutex);
pi += mypi;
pthread_mutex_unlock(&reduction_mutex);
```

每个线程先计算自己的局部积分 `mypi`，最后把结果累加到全局变量 `pi`。

由于 `pi` 是共享变量，所以累加时必须加锁。

这体现了共享内存并行程序的典型结构：

```text
局部计算尽量并行
共享结果更新必须互斥
```

## 13. 有界缓冲区问题的 Pthread 条件变量解

课件第 18 页给出生产者-消费者问题。

### 13.1 问题背景

有界缓冲区中：

```text
生产者 producer 产生数据放入缓冲区
消费者 consumer 从缓冲区取数据
缓冲区大小有限
```

要解决两个问题：

```text
缓冲区满时，生产者等待
缓冲区空时，消费者等待
```

------

### 13.2 生产者逻辑

生产者代码核心是：

```c
pthread_mutex_lock(&slot_lock);
while (nslots <= 0)
    pthread_cond_wait(&slots, &slot_lock);
nslots--;
pthread_mutex_unlock(&slot_lock);

put_item(i*i);
```

含义是：

```text
如果没有空槽位，就等待 slots 条件变量
有空槽后，占用一个槽位
放入数据
```

之后生产者增加已有 item 数量，并唤醒消费者：

```c
pthread_mutex_lock(&item_lock);
nitems++;
pthread_cond_signal(&items);
pthread_mutex_unlock(&item_lock);
```

------

### 13.3 消费者逻辑

消费者等待 item：

```c
pthread_mutex_lock(&item_lock);
while ((nitems <= 0) && !producer_done)
    pthread_cond_wait(&items, &item_lock);
```

如果没有数据且生产者未完成，就等待。

如果生产者完成且没有数据，就退出。

否则取出一个 item：

```text
nitems--
get_item(&myitem)
sum += myitem
```

然后释放一个槽位并唤醒生产者：

```c
pthread_mutex_lock(&slot_lock);
nslots++;
pthread_cond_signal(&slots);
pthread_mutex_unlock(&slot_lock);
```

------

### 13.4 这个例子说明什么

它体现了条件变量编程的基本模式：

```text
while 条件不满足：
    cond_wait

修改共享状态后：
    cond_signal 或 cond_broadcast
```

注意一定要用 `while` 而不是简单 `if`，因为线程被唤醒后仍然需要重新检查条件。

## 27. OpenMP 同步构造

课件第 40 页列出同步构造：

```text
MASTER
CRITICAL
BARRIER
ATOMIC
FLUSH
ORDERED
```

### 27.1 MASTER

只有主线程执行该区域。

不同于 `single`，`master` 不一定有隐式 barrier。

------

### 27.2 CRITICAL

临界区，同一时刻只有一个线程执行。

用于保护复杂共享变量更新。

------

### 27.3 BARRIER

所有线程必须到达 barrier 后才能继续。

------

### 27.4 ATOMIC

原子操作，适合简单共享变量更新，例如：

```fortran
x = x + 1
```

相比 critical，atomic 更轻量。

------

### 27.5 FLUSH

刷新线程可见的内存状态，保证共享变量的一致性可见。

------

### 27.6 ORDERED

按照循环迭代的顺序执行某段代码。

------

## 28. ORDERED 示例

课件第 41 页给出 ORDERED 示例。

```fortran
!$OMP PARALLEL
!$OMP DO ORDERED SCHEDULE(DYNAMIC)
DO I = LowBound, UpBound, Step
    CALL WORK(I)
END DO
!$OMP END PARALLEL

SUBROUTINE WORK(K)
!$OMP ORDERED
    WRITE(*,*) K
!$OMP END ORDERED
END
```

含义是：

```text
循环计算可以动态调度并行执行
但 ORDERED 区域中的 WRITE 必须按循环原顺序输出
```

这解决了“并行执行但输出顺序要保持一致”的问题。

## 30. OpenMP 的 Orphan 特性

### 30.1 Orphan 指令是什么

课件第 43 页定义：

> Orphan 制导语句是指那些在并行区域之外的制导语句。

也就是说，OpenMP 指令可以出现在被并行区域调用的子程序里，而不是必须直接写在 `parallel` 块内部。

------

### 30.2 为什么需要 Orphan

课件指出：

```text
为了便于支持粗粒度任务级并行
增加程序模块性
```

如果没有 Orphan，所有同步和控制语句必须依次写在并行区域内，程序会很长、很难模块化。

X3H5 不支持这一点，因此模块性较差。

------

### 30.3 Orphan 示例

第 44 页图中，外层有：

```fortran
!$OMP PARALLEL DEFAULT(PRIVATE) SHARED(...)
    call initialize_field(...)
    call compute_field(...)
    call compute_spectrum(...)
!$OMP END PARALLEL
```

而在被调用子程序内部，又出现：

```fortran
!$OMP DO
...
!$OMP END DO NOWAIT
```

这就是 orphan 指令：
`DO` 不在文本上的 parallel 区域内部，但运行时绑定到调用它的并行区域。

## 32. OpenMP 计算 π 的实例

课件第 47 页给出 OpenMP 计算 π 的 Fortran 示例。

### 32.1 核心公式

仍然是：

```text
π = ∫_0^1 4 / (1+x²) dx
```

通过循环累加近似积分。

------

### 32.2 OpenMP 并行部分

课件红框标出：

```fortran
!$OMP PARALLEL DO PRIVATE(x), SHARED(w)
!$OMP REDUCTION(+:sum)
do i = 1, n
    x = w * (i - 0.5d0)
    sum = sum + f(x)
enddo
```

这里：

```text
PARALLEL DO：并行循环
PRIVATE(x)：每个线程有自己的 x
SHARED(w)：w 被线程共享
REDUCTION(+:sum)：对 sum 做加法归约
```

### 32.3 为什么比 Pthreads 简洁

Pthreads 版本要手工：

```text
创建线程
计算线程编号
划分循环
加互斥锁
join 线程
```

OpenMP 版本只需要几条指令：

```text
PARALLEL DO
PRIVATE
SHARED
REDUCTION
```

这就是 OpenMP 的优势。

分别用于：

```

### 33.2 广播输入参数

红框中：

```

表示 root 进程把积分区间数 `n` 广播给所有进程。

### 33.3 局部计算和归约

每个进程计算自己的局部和：

```text

```

最后通过：

```fortran

```

把所有进程的局部结果加到 root 进程。

OpenMP 是共享内存：

```text
sum 可以通过 reduction 在共享地址空间中归约
```

MPI 是消息传递：

```text
必须显式 BCAST 和 REDUCE
```

所以 MPI 对程序员要求更高，但适合分布式内存系统。

## 37. 本章核心总结

### 37.1 本章讲什么

PC13.pdf 的核心是：

```text
共享存储系统编程
├── 共享内存并行标准需要规定执行模型、并行性、同步和运行环境
├── X3H5 是早期共享存储编程模型
├── Pthreads 是底层线程标准
└── OpenMP 是更高级、更易用的共享内存并行标准
```

------

### 37.2 最重要的理解

共享存储编程的核心矛盾是：

```text
通信很方便：线程可以直接访问共享变量
同步很困难：共享变量容易产生竞争和不一致
```

所以共享存储程序设计的重点是：

```text
哪些变量共享？
哪些变量私有？
哪里需要锁？
哪里需要 barrier？
哪里可以用 reduction？
哪里可以避免同步？
```

------

### 37.3 Pthreads 与 OpenMP 的区别

```text
Pthreads：显式创建线程、显式加锁、显式 join，控制力强但代码复杂。
OpenMP：用指令标注并行区域、并行循环、变量作用域和归约，代码简洁但控制较粗。
```

计算 π 的例子最能说明区别：

```text
Pthreads：手工 create/join/mutex/reduction
OpenMP：parallel do + private/shared + reduction
```
